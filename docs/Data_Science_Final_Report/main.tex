\section{Problem Definition \& Business Context}

\subsection{Business Problem}
Used car dealerships and online marketplaces receive large volumes of vehicle listings and trade-ins. Manually assigning each vehicle into a pricing/value tier is time-consuming and inconsistent across staff and branches. This project aims to automate the classification of incoming used cars into value tiers using historical market data.

\subsection{Stakeholders}

\begin{itemize}
  \item Primary: Used Car Dealerships (pricing and inventory teams)
  \item Secondary: Online Marketplaces (listing recommendations) and Insurance Underwriters (risk and valuation support)
\end{itemize}

\subsection{Problem Formulation}

This problem is formulated as a \textbf{classification task}, where the goal is to predict the \textbf{price/value tier} of a used car listing.

\textbf{Tier Definition:} Cars are grouped into \textbf{Budget}, \textbf{Mid-Range}, and \textbf{Luxury} tiers based on price.
\subsection{Justification for Classification}

\begin{itemize}
  \item The target is a discrete set of tiers (Budget / Mid-Range / Luxury)
  \item Tier outputs support operational decisions (pricing, procurement, and marketing)
  \item Classification enables threshold-based policies and consistent segmentation
\end{itemize}

\subsection{Business Impact}

\begin{itemize}
  \item Faster and more consistent inventory categorization
  \item Improved pricing strategy and margin optimization
  \item Better customer targeting (budget buyers vs. premium buyers)
\end{itemize}

\subsection{Data Integration Strategy}

\begin{itemize}
  \item \textbf{Vertical merge} the Kaggle dataset with scraped AutoScout24 listings by aligning both sources to a unified schema.
  \item Standardize fields and resolve naming differences (e.g., \texttt{kilometer} vs \texttt{mileage}).
  \item Deduplicate near-identical listings using matching keys such as \textbf{brand}, \textbf{model}, \textbf{kilometer}, \textbf{yearOfRegistration}, and \textbf{power}.
  \item Merge strategy is union rows after schema alignment.
\end{itemize}

\textbf{Challenges:}
\begin{itemize}
  \item Inconsistent naming and category sets across sources
  \item Translating the Kaggle dataset from German to English
  \item Missing or noisy scraped fields due to unstructured HTML
  \item Potential duplicates across sources
\end{itemize}

\subsection{Feature Transformation \& Justification}

\begin{itemize}
  \item \textbf{Numeric Imputation (median):} Missing numeric values are imputed using the median, which is robust to skewed distributions and outliers common in automotive pricing data.
  \item \textbf{Categorical Imputation (most\_frequent):} Mode imputation provides a stable default without introducing new categories, maintaining data consistency.
  \item \textbf{Scaling (StandardScaler):} Numeric features are standardized to zero mean and unit variance, improving optimization convergence and comparability for linear models.
  \item \textbf{One-Hot Encoding:} Applied to low-cardinality categorical variables to create binary indicator features, providing a strong baseline for mixed-type tabular data while reducing collinearity.
  \item \textbf{Frequency Encoding:} High-cardinality categorical variables (brand, model) are encoded by their frequency of occurrence in the training set. This prevents dimensionality explosion while preserving relative popularity information that correlates with vehicle value.
\end{itemize}

\subsection{Feature Engineering}
In addition to the core fields, we engineered the following domain-specific features to capture depreciation patterns and vehicle characteristics:

\begin{itemize}
  \item \textbf{vehicleAge}: Calculated as ($\text{current year} - \text{year of registration}$) to capture physical depreciation over time.
  \item \textbf{kmPerYear}: Annualized mileage ($\text{km} / (\text{age} + 1)$) representing usage intensity and wear patterns.
  \item \textbf{km\_power\_ratio}: Interaction between mileage and engine power ($\text{km} / (\text{power} + 1)$) capturing the relationship between usage and engine capacity. This emerged as the most important feature (22.9\% importance) in the final model.
  \item \textbf{log\_km}: Logarithmic transformation of mileage ($\log(1 + \text{km})$) to compress right-skewed outliers and normalize the distribution.
  \item \textbf{brand\_freq} \& \textbf{model\_freq}: Frequency encoding of brand and model based on their occurrence in the training data, capturing market presence and popularity effects on pricing.
\end{itemize}

\textbf{Feature Engineering Impact}: The engineered features dominated the top feature importance rankings, with \texttt{km\_power\_ratio} and \texttt{vehicleAge} being the two most predictive features. This validates that domain-informed feature engineering significantly improved model performance by capturing automotive depreciation dynamics.

\subsection{Feature Selection \& Justification}
We implemented and evaluated five feature selection strategies to identify the optimal subset of predictive features:

\textbf{Selection Strategies Evaluated}:
\begin{enumerate}
  \item \textbf{XGBoost Importance}: Model-based selection using XGBoost feature importance scores (threshold $\geq$ 0.01 or top 10 features).
  \item \textbf{Variance Threshold}: Removes low-variance features (threshold $\geq$ 0.01) that provide minimal discriminative power.
  \item \textbf{Correlation-based}: Selects features with strong absolute correlation to the target (threshold $\geq$ 0.05).
  \item \textbf{SelectKBest (f\_classif)}: Statistical selection using ANOVA F-values to identify top k features.
  \item \textbf{Mutual Information}: Information-theoretic selection capturing non-linear dependencies between features and target.
\end{enumerate}

\subsection{Train-Validation-Test Split Procedure}
We employed a three-way data splitting strategy to ensure robust model development and unbiased evaluation:

\begin{itemize}
  \item \textbf{Initial Split (Train + Val vs Test):} The full dataset was split 80/20 using stratified sampling to create a training/validation set (80\%) and a held-out test set (20\%). Stratification ensures the class distribution (Budget: 45.17\%, Mid-Range: 34.42\%, Luxury: 20.41\%) is preserved across all splits, which is crucial for imbalanced classification tasks.
  
  \item \textbf{Secondary Split (Train vs Validation):} The 80\% training/validation portion was further split 75/25 to create the final training set (60\% of total data) and validation set (20\% of total data), again using stratified sampling.
  
  \item \textbf{Validation Strategy:} Models are trained on the training set, hyperparameters are tuned using the validation set, and final performance is evaluated on the held-out test set. This prevents data leakage and ensures unbiased estimates of generalization performance.
\end{itemize}

\section{Exploratory Data Analysis}
 
\subsection{Overview \& Motivation}
 
Before committing to any modelling strategy, we conducted a structured
Exploratory Data Analysis (EDA) to build an evidence-based understanding
of what drives a vehicle's value tier.  Used-car pricing is not random:
dealerships and marketplaces rely on a handful of well-known market
signals---vehicle age, accumulated mileage, engine specification, brand
prestige, and body style---to position inventory.  Our EDA translated
these market intuitions into testable hypotheses, then evaluated each
hypothesis against the full dataset of \textbf{259,092 active listings}.
 
The insights obtained here served two purposes.  First, they validated
(or refined) our feature selection decisions.  Second, and crucially for
modelling, they revealed that the relationship between the predictors and
the price tier is \emph{non-linear and interaction-driven}---a finding
that directly motivated the choice of tree-based ensemble methods over
simpler parametric models.

\subsection{Hypothesis-Driven Analysis}
 
Each hypothesis below was formed from domain knowledge about the
European used-car market and then tested visually against the cleaned
dataset.  Where a hypothesis was \emph{confirmed}, the plot is used as
evidence to retain the corresponding feature.  Where it was only
\emph{partially confirmed}, the nuance is noted and informs how the

% ── H1 ────────────────────────────────────────────────────────
\subsubsection{Hypothesis 1 --- Vehicle Age and the Budget Tier}
 
\textbf{Hypothesis:} Older vehicles are more likely to be classified as
\emph{Budget}.
 
\textbf{Business rationale.}  From a dealership perspective, vehicle age
is one of the first proxies used during appraisal.  Depreciation in the
European market is steep in the first three to five years and then
flattens; by the time a car is more than ten years old it has typically
exhausted most of its resale premium.  A systematic age-to-tier
relationship would mean that \texttt{yearOfRegistration} (or its
engineered derivative \texttt{vehicleAge}) is a reliable, low-cost
signal for automated tier assignment.
 
\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.82\linewidth]{price-tier_vs_yearOfRegistration.png}
  \caption{Tier vs.\ Vehicle Age --- box-plots of year of registration
           by price tier.}
  \label{fig:tier_age}
\end{figure}
 
\textbf{Finding.}  Figure~\ref{fig:tier_age} confirms the hypothesis.
The interquartile range for the \emph{Budget} tier is centred around
the year 2000, while the \emph{Mid-Range} and \emph{Luxury} boxes shift
progressively towards more recent registration years, with \emph{Luxury}
medians concentrated after 2008.  The tail of very old outliers
(pre-1960) appears across all tiers, reflecting a small number of
classic-car listings, but the core bulk of budget inventory is
unmistakably older.  This confirms that \texttt{yearOfRegistration} is a
meaningful predictor and should be retained---or re-expressed as
\texttt{vehicleAge}---in the feature set.
 
% ── H2 ────────────────────────────────────────────────────────
\subsubsection{Hypothesis 2 --- Accumulated Mileage and Budget Classification}
 
\textbf{Hypothesis:} Higher accumulated mileage pushes a vehicle towards
the \emph{Budget} tier.
 
\textbf{Business rationale.}  Mileage is perhaps the single most
scrutinised figure in a used-car transaction.  High-mileage vehicles
carry greater mechanical risk, command lower trade-in values, and are
routinely priced at a discount.  Conversely, low-mileage luxury vehicles
retain premium pricing because they signal careful, limited use.  If
mileage reliably separates tiers, it justifies its inclusion as a
first-class feature.
 
\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.82\linewidth]{price_tier_vs_kilometer.png}
  \caption{Tier vs.\ Mileage Intensity --- box-plots of accumulated
            kilometres by price tier.}

\end{figure}
 
\textbf{Finding.}  Figure~\ref{fig:tier_km} reveals a more nuanced
picture than expected.  The \emph{Budget} tier shows an extremely
compressed interquartile range sitting high at roughly 150,000~km,
whereas \emph{Mid-Range} vehicles span a wide band from near zero to
225,000~km and \emph{Luxury} vehicles cluster at a lower median of
approximately 90,000~km with a similarly wide spread.  The hypothesis is
\emph{partially confirmed}: high mileage is indeed concentrated in the
budget segment, but the luxury tier does not display the very low mileage
that a simplistic view would predict---many luxury cars in this market
are pre-owned daily drivers, not garage queens.  The practical implication
is that mileage alone does not linearly separate the tiers; it interacts
with brand and vehicle age, reinforcing the need for a model capable of
capturing such interactions.
 
% ── H3 ────────────────────────────────────────────────────────
\subsubsection{Hypothesis 3 --- Engine Power and the Luxury Tier}
 
\textbf{Hypothesis:} Greater engine power increases the probability of a
\emph{Luxury} classification.
 
\textbf{Business rationale.}  Horsepower is a premium attribute in the
European used-car market.  High-output engines are disproportionately
found in sports cars, executive saloons, and SUVs---vehicle categories
that command luxury pricing.  A strong power-to-tier relationship would
validate engine power as a differentiating feature and support its use
in automated valuation.

\textbf{Finding.}  Figure~\ref{fig:tier_power} strongly confirms the

extending towards 250~hp.  The \emph{Luxury} violin is the broadest of

high-performance sports and super cars.  The multi-modal, notched shape
of all three violins also confirms that power is not normally distributed

exploit this feature effectively.
 
% ── H4 ────────────────────────────────────────────────────────
\subsubsection{Hypothesis 4 --- Brand Prestige and Luxury Market Share}
 
\textbf{Hypothesis:} Premium brands (BMW, Mercedes-Benz, Audi) carry a
disproportionately high share of \emph{Luxury} listings.
 
\textbf{Business rationale.}  Brand is arguably the strongest single
heuristic that human appraisers apply.  A \emph{BMW 7 Series} or
\emph{Mercedes-Benz S-Class} is almost always a luxury vehicle regardless
of age or mileage, whereas a \emph{Volkswagen Polo} is almost never one.
Quantifying brand-level tier composition allows dealerships to set
brand-specific pricing floors and ceilings, and it validates the inclusion
of a brand-derived feature---either as a raw categorical variable or as a
frequency-encoded representation of its luxury share.

luxury premium is not evenly distributed across the brand ladder but is
concentrated in a handful of premium manufacturers, making it ideal

\subsubsection{Hypothesis 5 --- Vehicle Body Type and Tier Composition}
 
\textbf{Hypothesis:} Vehicle body type influences tier classification,
with certain body styles (convertibles, coupes) skewing towards
\emph{Luxury} and others (compacts, vans) towards \emph{Budget}.
 
\textbf{Business rationale.}  Body style encodes both the functional
purpose and the target customer of a vehicle.  Convertibles and coupes
are predominantly lifestyle or sports vehicles and are priced accordingly.
Compact hatchbacks and vans are utility-oriented and occupy the budget end
of the market.  If body style shows a statistically meaningful tier
distribution, it can serve as a lightweight but powerful feature in the
classifier.

warrants explicit inclusion in the model.

\subsection{From EDA Insights to Model Selection}
 
The five hypotheses, taken together, paint a coherent picture of the
used-car value market: \emph{price tier is determined by a combination
of physical condition signals (age, mileage), performance attributes
(engine power), brand prestige, and vehicle category---and these factors
interact in complex, non-linear ways.}
 
Several specific observations from the EDA ruled out simpler modelling
approaches:
 
\begin{itemize}
  \item \textbf{Non-linearity.}  The multi-modal violin plots for engine
        power and the compressed, asymmetric mileage distributions
        demonstrate that no single linear boundary can cleanly separate
        the tiers.
  \item \textbf{Feature interactions.}  A high-mileage vehicle is
        \emph{budget} if it is a Ford, but may still be \emph{mid-range}
        if it is an Audi with a large engine.  Mileage, brand, and power
        must be considered jointly, not independently.
  \item \textbf{Categorical heterogeneity.}  Both brand and vehicle type
        exhibit tier distributions that cannot be captured by a simple
        ordinal encoding; the relationship is categorical and
        context-dependent.
  \item \textbf{Class imbalance.}  With a luxury share of 20.4\%, the
        target is moderately imbalanced, requiring a model that does not
        trivially favour the majority class.
\end{itemize}
 
These characteristics collectively point to \textbf{tree-based ensemble
models}---specifically gradient-boosted decision trees---as the natural
modelling choice.  Decision trees handle non-linearity and feature
interactions natively through recursive binary splits; boosting reduces
variance and corrects misclassification errors iteratively; and the
ensemble structure is robust to the categorical heterogeneity observed
across brands and body types.  Logistic Regression, as expected from
these EDA insights, was subsequently evaluated and found to underfit
significantly (F1 $\approx 0.66$), confirming that a non-linear model
was essential.  \textbf{XGBoost} ultimately delivered the best
performance, achieving a Test F1 of \textbf{0.859} with a severe-error
rate of only \textbf{0.22\%}, validating both the EDA-driven feature set
and the model family selection.

\section{Model Development}
We evaluated multiple model architectures to address the imbalanced 3-class target variable (budget, mid-range, luxury). The evaluation included 5 families of models across 3 data balancing strategies (none, SMOTE, undersampling).

\subsection{Model Candidates and Rationale}
\begin{itemize}
    \item \textbf{Baseline}: A \texttt{DummyClassifier} (most\_frequent strategy) was used to establish the minimum expected performance. The baseline achieved F1=0.207, confirming the dataset is learnable.
    \item \textbf{Linear Models}: \texttt{LogisticRegression} (saga solver) and \texttt{LinearSVC} were tested as strong, fast baselines for tabular data. These models struggled with the non-linear relationships in the data (F1 $\approx$ 0.66).
    \item \textbf{Single Tree}: \texttt{DecisionTreeClassifier} was included as an interpretable non-linear baseline, achieving F1=0.830.
    \item \textbf{Ensembles}: \texttt{RandomForestClassifier} and \texttt{ExtraTreesClassifier} were utilized to natively capture non-linearities and feature interactions (F1=0.822-0.826).
    \item \textbf{Gradient Boosting}: \texttt{XGBClassifier} was selected as the state-of-the-art candidate for handling complex tabular patterns, achieving the best performance (F1=0.859).
\end{itemize}

\subsection{Hyperparameter Search Strategy}
\begin{itemize}
    \item \textbf{Search Method}: \texttt{ParameterSampler} with randomized search
    \item \textbf{Optimization Metric}: Macro F1 score (treats all classes equally)
    \item \textbf{Iterations per Model}: 5-20 iterations depending on model complexity
    \item \textbf{Key Parameters Tuned}:
    \begin{itemize}
        \item XGBoost: n\_estimators (300-800), max\_depth (4-10), learning\_rate (0.01-0.1), subsample, colsample\_bytree
        \item Random Forest: n\_estimators (100-300), max\_depth (None, 10-30), min\_samples\_split
        \item Logistic Regression: C (0.01-10.0)
    \end{itemize}
\end{itemize}

This critical business metric measures Budget $\leftrightarrow$ Luxury errors, representing the highest-risk classification mistakes. At \textbf{0.28\%}, the model maintains strong safety despite the increased luxury recall:
\begin{itemize}
    \item Approximately 145 severe errors out of 51,819 predictions (compared to 114 with non-resampled model)
    \item Still far lower than Logistic Regression (2.1-2.7\%)
    \item Well below the 0.5\% business safety threshold
    \item Validates the model for automated pricing decisions with appropriate confidence intervals
\end{itemize}

\textbf{Trade-off Analysis}: The 0.06 percentage point increase in severe errors (from 0.22\% to 0.28\%) is more than offset by the 4.4 percentage point gain in luxury recall (from 86.1\% to 90.5\%), representing a favorable business risk-reward ratio.

\subsection{Resampling Strategy Impact}
\begin{itemize}
    \item \textbf{Undersampling (Selected)}: \textbf{Highest luxury recall (90.53\%)} with acceptable trade-offs. F1=0.8529 (-0.6pp), severe error=0.28\% (+0.06pp). \textbf{Deployed for production} due to business priority on premium vehicle detection.
    \item \textbf{SMOTE}: Good luxury recall (88.57\%, +2.5pp) with F1=0.8560 (-0.3pp). Balanced middle ground option.
    \item \textbf{No Resampling}: Best overall F1 (0.8590) and lowest severe error (0.22\%). Luxury recall limited to 86.11\%. Alternative for conservative deployments.
\end{itemize}

\textbf{Recommendation}: Deploy \textbf{XGBoost with undersampling} for production to maximize luxury vehicle detection (90.5\% recall). The 470 additional luxury cars detected per 10,000 listings significantly outweigh the minor decreases in overall F1 and slight increase in severe errors. Keep the non-resampled variant available for use cases prioritizing overall accuracy over luxury detection.

\section{Code Automation}
To ensure reproducibility and streamline the development workflow, we utilized \texttt{Make} as our primary automation tool. The \texttt{Makefile} is configured to automate various critical stages of the machine learning pipeline, abstracting complex commands into simple targets:
\begin{itemize}
    \item \textbf{Data Acquisition \& Preprocessing:} Targets such as \texttt{make merge}, \texttt{make validate\_raw}, and \texttt{make clean} automate the execution of Python and Bash scripts for merging sources, validating, and cleaning data.
    \item \textbf{Data Validation:} The \texttt{make validate\_raw} and \texttt{make validate\_preprocessed} commands ensure data integrity at different pipeline stages.
    \item \textbf{Code Quality:} We enforce clean code practices by automating formatting with Black (\texttt{make format}) and linting with Flake8 (\texttt{make lint}).
    \item \textbf{Testing:} The \texttt{make test} target executes the Pytest suite, generating detailed terminal and HTML coverage reports.
    \item \textbf{End-to-End Execution:} The \texttt{make pipeline} target runs the complete data preparation and quality check sequence (merge, validate, clean, check, and delete) sequentially, while \texttt{make check} bundles formatting and linting.
\end{itemize}
This structured automation simplifies team collaboration and integrates seamlessly with our continuous integration workflows.

\section{Continuous Integration}
We implemented a Continuous Integration (CI) pipeline using GitHub Actions to automatically enforce code quality, run testing suites, and ensure code robustness on every update. The pipeline is configured to trigger automatically on any \texttt{push} or \texttt{pull\_request} to the \texttt{main} branch.

The workflow is executed on an \texttt{ubuntu-latest} runner and consists of the following automated steps:
\begin{itemize}
    \item \textbf{Environment Setup:} Checks out the repository, initializes Python 3.14, and installs the Poetry package manager (version 2.1.1) to handle dependencies in an isolated virtual environment.
    \item \textbf{Dependency Management:} Automatically generates the \texttt{poetry.lock} file and installs all project dependencies without interaction.
    \item \textbf{Static Code Analysis (Linting):} Executes Flake8 via Poetry against the \texttt{src/} and \texttt{tests/} directories to ensure the codebase maintains clean code practices and formatting standards.
    \item \textbf{Automated Testing \& Coverage:} Runs the comprehensive Pytest suite, generating both terminal and HTML coverage reports (\texttt{--cov-report=term-missing --cov-report=html}) to verify code correctness and identify untested logic.
\end{itemize}
This CI configuration prevents breaking changes from being merged into the main branch and guarantees that all integrated code meets the required quality thresholds outlined in the project requirements.

\section{Interactive Dashboard}

\subsection{Purpose \& Business Value}
To ensure the machine learning model delivers practical value, an interactive web dashboard was developed. It bridges the gap between the complex predictive model and non-technical stakeholders.\\
\\ It provides an intuitive interface that allows users to instantly classify incoming vehicle inventory into pricing tiers without requiring any programming knowledge.

\subsection{Technology Stack}
\begin{itemize}
  \item \textbf{Web Framework:} Streamlit (chosen for its rapid UI development and seamless integration with Python data science libraries).
  \item \textbf{Visualization:} Plotly Express (utilized for rendering interactive, stakeholder-friendly charts).
  \item \textbf{Backend/Inference:} Pandas (for on-the-fly data manipulation) and the saved Machine Learning model (for real-time predictions).
\end{itemize}

\subsection{Key Dashboard Components}
The dashboard is structurally designed to guide the stakeholder from high-level market metrics down to specific vehicle predictions and model explainability.

\begin{enumerate}
  \item \textbf{Executive Summary Metrics:}
  The top of the dashboard displays dynamic, high-level KPIs, including the total volume of historical vehicles, average market price, and the current luxury market share. This provides immediate macro-level context.

  \item \textbf{Interactive "What-If" Controls:}
  A sidebar interface allows users to input specific vehicle parameters (Brand, Engine Power, Mileage, Registration Year, and Transmission Type). This acts as a sandbox for stakeholders to test hypothetical inventory scenarios.

  \item \textbf{Live Prediction Engine:}
  The core of the application captures the user's input, automatically replicates the exact feature engineering steps from the training phase (e.g., calculating \texttt{vehicleAge} and applying frequency encodings), and queries the trained model to output a real-time Value Tier prediction (Budget, Mid-Range, or Luxury).

  \item \textbf{Model Insights \& Explainability:}
  To prevent the model from acting as a "black box" and to build trust with business users, the dashboard features a dynamic Feature Importance chart. This visualizes exactly which vehicle attributes have the strongest influence on the algorithm's pricing decisions.

    \item \textbf{Model Selection \& Comparasions:}
    A comparison section allows users to view different model versions (e.g., with or without resampling) to see how their performance and predictions vary.
\end{enumerate}

\section{Limitations \& Future Work}
\subsection{Limitations}
We acknowledge the following limitations in our current approach:
\begin{itemize}
  \item Data quality issues such as missing values and potential outliers may impact model performance.
  \item Class imbalance, even after balancing techniques, may still affect the model's generlization.
  \item The model may not capture all relevant features influencing car pricing, such as regional market differences or economic factors.
  \item The current model is based on historical data and needs retraining on future market changes.
\end{itemize}

\subsection{Future Work}
Future work could include:
\begin{itemize}
  \item  Investigating the impact of external factors (e.g., economic conditions, seasonal trends) on car pricing and incorporating them into the model.
  \item Exploring advanced modeling techniques such as ensemble methods or deep learning architectures
  \item  Evaluating model performance across different market segments and regions to ensure robustness.
  \item Conducting a more thorough error analysis to identify specific areas for improvement.
  \item Implementing a feedback loop from the deployed model to continuously improve performance based on real-world data and user interactions.
\end{itemize}
